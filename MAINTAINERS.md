## 维护者职责

### 通用职责

所有维护者应履行以下职责：

#### 1. 代码审查

- 及时审查 Pull Requests
- 提供建设性反馈
- 确保代码质量符合标准
- 验证测试覆盖率
- 检查文档更新

#### 2. 问题管理

- 分类和标记 Issues
- 回复用户问题
- 指导用户解决问题
- 识别重复问题
- 维护问题模板

#### 3. 版本发布

- 准备发布计划
- 更新 CHANGELOG
- 创建发布标签
- 发布公告
- 监控发布后问题

#### 4. 安全管理

- 处理安全漏洞报告
- 协调安全补丁
- 发布安全公告
- 申请 CVE（如需要）
- 更新安全文档

#### 5. 社区管理

- 促进健康讨论
- 调解冲突
- 欢迎新贡献者
- 组织社区活动
- 维护社区氛围

#### 6. 文档维护

- 保持文档更新
- 改进文档质量
- 添加示例和教程
- 翻译文档（如适用）

### 专门职责

#### 技术负责人

- 制定技术方向
- 架构设计决策
- 技术债务管理
- 性能优化
- 安全架构

#### 发布经理

- 版本规划
- 发布流程执行
- 变更日志维护
- 发布公告撰写

#### 社区经理

- 社区增长
- 用户反馈收集
- 贡献者培养
- 活动组织

### GitHub 权限

| 权限 | 说明 |

| Write | 直接推送到仓库 |

### 责任与权限对等

拥有权限意味着承担相应责任：

- **谨慎合并**: 确保代码质量
- **及时响应**: 处理分配的问题
- **透明决策**: 公开决策理由
- **团队协作**: 重大决策与团队协商

## 成为维护者

### 贡献者路径

```
用户 → 贡献者 → 活跃贡献者 → 核心贡献者 → 维护者
```

### 评估标准

我们根据以下标准评估潜在维护者：

#### 1. 持续性

- 持续贡献至少 **3 个月**
- 每月至少 **3 次有效贡献**
- 证明长期投入意愿

#### 2. 代码质量

- 提交的代码质量高
- 测试覆盖率达标
- 代码审查反馈积极
- 遵循项目规范

#### 3. 社区参与

- 积极参与 Discussions
- 帮助其他用户
- 审查他人代码
- 参与设计讨论

#### 4. 文化契合

- 理解项目理念
- 维护友好社区氛围
- 尊重不同观点
- 建设性沟通

### 提名流程

1. **自我提名或他人提名**: 在 Discussions 发起提名
2. **社区反馈**: 社区讨论候选人
3. **维护者评估**: 现有维护者评估候选人
4. **试考察期**: 1 个月的考察期
5. **正式任命**: BDFL 批准后正式任命

### 新维护者入职

新维护者将获得：

- GitHub 权限提升
- 维护者指南访问
- 私有沟通渠道邀请
- 欢迎公告

## 维护者行为准则

### 期望行为

维护者应：

1. **以身作则**: 遵守行为准则，展示最佳实践
2. **尊重**: 尊重所有贡献者，无论经验水平
3. **及时响应**: 在合理时间内回复问题和 PR
4. **建设性**: 提供建设性反馈，帮助改进
5. **透明**: 公开决策过程和理由
6. **协作**: 与其他维护者和社区协作
7. **学习**: 持续学习新技术和最佳实践

### 利益冲突

维护者应避免利益冲突：

- 公开任何可能影响决策的利益关系
- 在涉及自身利益的决策中回避
- 优先考虑项目利益而非个人利益

### 时间承诺

维护者应承诺：

- 每周至少 **2 小时** 用于项目维护
- 预报长期缺席时间
- 安排缺席期间的职责代理

## 维护者轮换

### 活跃状态

维护者分为两种状态：

- **活跃**: 正常履行维护职责
- **休假**: 暂时无法履行职责

### 休假流程

1. 提前通知团队
2. 说明休假时间
3. 安排职责代理
4. 返回后重新激活

---

## 卸任流程

### 自愿卸任

维护者可以随时自愿卸任：

1. 通知 BDFL 和维护团队
2. 说明卸任原因（可选）
3. 协助知识转移
4. 公开感谢

### 自动卸任

以下情况可能导致自动卸任：

- **长期不活跃**: 6 个月无任何贡献
- **严重违规**: 违反行为准则且无法纠正
- **利益冲突**: 无法解决的严重利益冲突

### 卸任后

- 保留贡献者荣誉
- 可重新申请成为维护者
- 如有需要，移除敏感访问权限
